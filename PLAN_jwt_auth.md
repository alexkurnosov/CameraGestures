# Plan: Per-Device JWT Authentication

## Overview

Before the training server is deployed to a public VPS it needs an authentication layer. Right now every endpoint is completely open — anyone who knows the IP can upload training data, trigger model training, or download the model. The plan adds per-device JWT tokens: each iOS device registers once with a shared secret, gets a signed token, and includes that token in every subsequent request.

`/health` remains open so Docker's health-check probe keeps working without credentials.

---

## Steps

---

### Step 1 — Add `python-jose` to server dependencies

**File:** `server/requirements.txt`

**Short description**
Install the Python library that creates and validates JWT tokens.

**Simple explanation**
JWT tokens are signed strings that prove identity without the server having to store session state. `python-jose` is the standard library for producing and reading them in Python. Without it there is no way to sign or verify tokens.

**Detailed technical explanation**
Add the line:
```
python-jose[cryptography]>=3.3
```
The `[cryptography]` extra installs the `cryptography` backend (instead of the lighter `pycryptodome`), which is required for the HS256 HMAC-SHA256 algorithm we use. The library exposes `jose.jwt.encode` and `jose.jwt.decode` — the two functions used in Step 5.

**Why we need it**
Without this package the server has no ability to create or validate JWT tokens. It is the foundation that all subsequent auth steps depend on.

---

### Step 2 — Add auth settings to server config

**File:** `server/config.py`

**Short description**
Add four new configuration fields to `Settings` for JWT signing and device registration.

**Simple explanation**
The server needs to know: what secret to use when signing tokens, what algorithm to use, how long tokens should last, and what secret phrase an iOS device must present to be allowed to register. All of these come from environment variables so nothing sensitive is hardcoded.

**Detailed technical explanation**
Add four fields to the existing `Settings(BaseSettings)` class:

```python
jwt_secret: str            # no default — server refuses to start if missing
jwt_algorithm: str = "HS256"
jwt_expire_days: int = 365
registration_token: str    # no default — server refuses to start if missing
```

`jwt_secret` is used as the HMAC key when signing JWTs. It should be a long random string (e.g. `openssl rand -hex 32`). `registration_token` is the pre-shared secret the iOS device must present to obtain its first JWT — it functions like a one-time invite code that gates who can register. Pydantic will raise a `ValidationError` at startup if either required field is absent from the environment, preventing the server from running misconfigured.

**Why we need it**
Centralising secrets in `Settings` means they are injected through `.env` / environment variables and never committed to source control. The no-default constraint acts as a safety net: a misconfigured deploy fails loudly rather than silently accepting all requests.

### Step 3 — Add `devices` table to the database

**File:** `server/database.py`

**Short description**
Add a table that records which device IDs have been registered.

**Simple explanation**
When an iOS device registers for the first time the server needs somewhere to remember it. This table holds the device's identifier and the time it first registered. It also makes future features possible, like revoking a specific device's access.

**Detailed technical explanation**
Add the following SQLAlchemy `Table` alongside the existing `examples_table` and `models_table`:

```python
devices_table = Table(
    "devices", metadata,
    Column("device_id", String, primary_key=True),
    Column("registered_at", Float, nullable=False),
)
```

`device_id` is a string UUID generated by the iOS app and used as the `sub` claim in the JWT. `registered_at` stores the Unix timestamp of first registration. Update `init_db()` to include this table in the `CREATE TABLE IF NOT EXISTS` pass so it is created automatically on startup alongside the existing tables.

**Why we need it**
Persisting device registrations enables auditing (which devices have connected), and in the future it allows the server operator to revoke a specific device by deleting its row. Without this table registration would be entirely stateless, meaning revocation would require rotating the whole JWT secret.

### Step 4 — Add auth Pydantic models

**File:** `server/models.py`

**Short description**
Add the request and response shapes for the registration endpoint.

**Simple explanation**
FastAPI uses Pydantic models to describe what JSON it expects to receive and what it will send back. We need to define what a registration request looks like (device ID + secret) and what the server replies with (the JWT token).

**Detailed technical explanation**
Add two classes:

```python
class DeviceRegisterRequest(BaseModel):
    device_id: str
    registration_token: str

class TokenResponse(BaseModel):
    access_token: str
    token_type: str = "bearer"
```

`DeviceRegisterRequest` is the JSON body the iOS app posts to `/auth/register`. `TokenResponse` is what the server returns — it follows the OAuth2 bearer token convention so the token format is recognisable to standard tooling (curl, Postman, OpenAPI UI).

**Why we need it**
FastAPI auto-validates incoming JSON against the request model (rejects malformed payloads with a clear `422` error) and auto-generates OpenAPI docs from the response model. Defining them here keeps all data shapes in one place, consistent with the rest of the codebase.

### Step 5 — Create `server/auth.py` (JWT utilities)

**File:** `server/auth.py` *(new)*

**Short description**
A small module with two things: a function that creates JWTs, and a FastAPI dependency that validates them.

**Simple explanation**
This is the engine of the auth system. One function stamps a token with a device ID and an expiry date. The other function is plugged into every protected endpoint and checks that the incoming token is genuine and not expired.

**Detailed technical explanation**
The module contains:

1. **`create_access_token(device_id: str) -> str`**
   Builds a payload `{"sub": device_id, "exp": datetime.utcnow() + timedelta(days=settings.jwt_expire_days)}` and calls `jose.jwt.encode(payload, settings.jwt_secret, algorithm=settings.jwt_algorithm)`. Returns the compact serialised JWT string.

2. **`get_current_device` dependency**
   Declared as:
   ```python
   oauth2_scheme = OAuth2PasswordBearer(tokenUrl="auth/register")

   async def get_current_device(token: str = Depends(oauth2_scheme)) -> str:
       try:
           payload = jwt.decode(token, settings.jwt_secret, algorithms=[settings.jwt_algorithm])
           device_id: str = payload.get("sub")
           if device_id is None:
               raise credentials_exception
           return device_id
       except JWTError:
           raise HTTPException(status_code=401, detail="Invalid or expired token")
   ```
   `OAuth2PasswordBearer` tells FastAPI (and OpenAPI docs) where the token comes from and causes it to extract the `Authorization: Bearer <token>` header automatically. Any endpoint that declares `Depends(get_current_device)` will refuse the request with `401` if the token is missing, malformed, or expired.

**Why we need it**
Centralising token creation and validation here means there is one place to change the algorithm, the expiry, or the secret rotation logic. It also keeps the router files clean — they declare the dependency but contain none of the cryptographic details.

### Step 6 — Create `server/routers/auth.py` (registration endpoint)

**File:** `server/routers/auth.py` *(new)*

**Short description**
A single endpoint: `POST /auth/register`. It checks the pre-shared secret, records the device, and returns a JWT.

**Simple explanation**
This is the door every new iOS device knocks on. The device says "I know the secret passphrase and here is my ID". If the passphrase is correct the server hands back a token. If it's wrong the server refuses. After this first exchange the device never needs to send the passphrase again — it uses the token instead.

**Detailed technical explanation**
```python
@router.post("/auth/register", response_model=TokenResponse)
async def register_device(body: DeviceRegisterRequest, db=Depends(get_db)):
    # Constant-time comparison to prevent timing attacks
    if not hmac.compare_digest(body.registration_token, settings.registration_token):
        raise HTTPException(status_code=403, detail="Invalid registration token")

    # Upsert: if device already exists, update registered_at
    await db.execute(
        insert(devices_table)
        .values(device_id=body.device_id, registered_at=time.time())
        .on_conflict_do_update(
            index_elements=["device_id"],
            set_={"registered_at": time.time()}
        )
    )
    await db.commit()

    return TokenResponse(access_token=create_access_token(body.device_id))
```

Key details:
- **`hmac.compare_digest`**: compares the two strings in constant time. A naive `==` comparison short-circuits on the first differing byte, leaking timing information that could be exploited to guess the token character by character.
- **Upsert semantics**: calling `/auth/register` a second time with the same `device_id` is safe — it refreshes the registration timestamp and issues a fresh token. This is the mechanism the "Re-register Device" button in iOS Settings uses.
- **No rate limiting in this step**: the endpoint is open (no JWT required to call it). Rate limiting should be added at the reverse-proxy (nginx/Caddy) level on the VPS.

**Why we need it**
This is the only way a new device obtains a JWT. Without it the entire auth chain has no entry point.

### Step 7 — Register the auth router in `server/main.py`

**File:** `server/main.py`

**Short description**
Tell FastAPI that the `/auth` routes exist.

**Simple explanation**
FastAPI only knows about a set of endpoints if they are explicitly included. This step plugs the new auth router into the application so `/auth/register` becomes reachable.

**Detailed technical explanation**
Add two lines to `main.py`:
```python
from routers import auth            # new import

app.include_router(auth.router)     # alongside existing include_router calls
```
The existing `/health` endpoint defined directly on `app` remains unauthenticated — it has no `Depends(get_current_device)` in its signature.

**Why we need it**
Without this include the registration endpoint simply does not exist from the outside, making the whole auth system unreachable.

### Step 8 — Protect all existing business endpoints

**Files:** `server/routers/examples.py`, `server/routers/training.py`, `server/routers/model.py`

**Short description**
Add a single line to every existing endpoint so it requires a valid JWT.

**Simple explanation**
At this point the auth machinery exists but the existing endpoints don't know about it — they are still open. This step locks all of them behind the token check with the smallest possible code change.

**Detailed technical explanation**
For every `@router.get / @router.post / @router.delete` function, add one parameter to the signature:
```python
_: str = Depends(get_current_device)
```
The underscore signals that the value is intentionally unused — we only care that the dependency runs (validates the token) and raises `401` if it fails. No other logic in any of these files changes. Example, in `examples.py`:
```python
@router.post("/examples", response_model=UploadExampleResponse)
async def upload_example(
    payload: TrainingExamplePayload,
    _: str = Depends(get_current_device),   # ← add this
):
    ...
```

**Why we need it**
Without this step the auth system exists but protects nothing. All the work in Steps 1–7 would be bypassed by simply calling the existing endpoints directly.

### Step 10 — Create `TokenStorage.swift` (iOS Keychain wrapper)

**File:** `iOS/ModelTraining/ModelTraining/ModelTraining/Networking/TokenStorage.swift` *(new)*

**Short description**
A small Swift class that saves, loads, and deletes the JWT from the iOS Keychain.

**Simple explanation**
The Keychain is iOS's built-in secure storage for secrets. It is encrypted and survives app reinstalls (unlike UserDefaults). We store the JWT there so it isn't sitting in plain-text storage that could be read by other processes or exposed in a device backup.

**Detailed technical explanation**
The class wraps three Keychain operations under the service `"GestureModelAPI"` and account `"jwt_token"`:

```swift
final class TokenStorage {

    private let service = "GestureModelAPI"
    private let account = "jwt_token"

    func save(_ token: String) {
        let data = Data(token.utf8)
        let query: [CFString: Any] = [
            kSecClass: kSecClassGenericPassword,
            kSecAttrService: service,
            kSecAttrAccount: account,
        ]
        SecItemDelete(query as CFDictionary)  // delete existing before inserting
        let attributes = query.merging([kSecValueData: data]) { $1 }
        SecItemAdd(attributes as CFDictionary, nil)
    }

    func load() -> String? {
        let query: [CFString: Any] = [
            kSecClass: kSecClassGenericPassword,
            kSecAttrService: service,
            kSecAttrAccount: account,
            kSecReturnData: true,
            kSecMatchLimit: kSecMatchLimitOne,
        ]
        var result: AnyObject?
        guard SecItemCopyMatching(query as CFDictionary, &result) == errSecSuccess,
              let data = result as? Data else { return nil }
        return String(decoding: data, as: UTF8.self)
    }

    func delete() {
        let query: [CFString: Any] = [
            kSecClass: kSecClassGenericPassword,
            kSecAttrService: service,
            kSecAttrAccount: account,
        ]
        SecItemDelete(query as CFDictionary)
    }
}
```

No special Keychain Sharing entitlement is required — basic `kSecClassGenericPassword` operations work in any app.

**Why we need it**
Storing the JWT in UserDefaults would leave it in a plain-text plist file. The Keychain provides OS-level encryption and access control. This wrapper gives the rest of the code a clean interface without scattering Keychain boilerplate across multiple files.

### Step 11 — Update `GestureModelAPIClient.swift` (auth logic)

**File:** `iOS/ModelTraining/ModelTraining/ModelTraining/Networking/GestureModelAPIClient.swift`

**Short description**
Add device registration logic and attach the JWT to every outgoing request.

**Simple explanation**
This is where the iOS side does the work. Before sending any real request the client checks whether it already has a token. If it does, it attaches it to the request header. If it doesn't, it first calls the registration endpoint to get one. If the server says the token is invalid it clears the stored token and reports an error.

**Detailed technical explanation**

**New stored state** (added to the class):
```swift
private static let registrationTokenKey = "GestureModelAPIClient.registrationToken"
private static let deviceIdKey          = "GestureModelAPIClient.deviceId"

@Published var registrationToken: String {
    didSet { UserDefaults.standard.set(registrationToken, forKey: Self.registrationTokenKey) }
}

private var deviceId: String   // generated once, stored in UserDefaults

private let tokenStorage = TokenStorage()
```

`deviceId` is generated with `UUID().uuidString` on first launch and persisted in UserDefaults. It is stable per app installation.

**`ensureAuthenticated()` method**:
```swift
func ensureAuthenticated() async throws {
    guard tokenStorage.load() == nil else { return }   // already have a token

    guard !registrationToken.isEmpty else {
        throw APIError.missingRegistrationToken
    }

    var request = URLRequest(url: baseURL.appendingPathComponent("auth/register"))
    request.httpMethod = "POST"
    request.setValue("application/json", forHTTPHeaderField: "Content-Type")
    let body = ["device_id": deviceId, "registration_token": registrationToken]
    request.httpBody = try encoder.encode(body)

    let (data, response) = try await session.data(for: request)
    guard let http = response as? HTTPURLResponse, (200..<300).contains(http.statusCode) else {
        let detail = (try? JSONDecoder().decode(ServerErrorDetail.self, from: data))?.detail ?? "registration failed"
        throw APIError.httpError(statusCode: (response as? HTTPURLResponse)?.statusCode ?? 0, detail: detail)
    }
    let tokenResponse = try decoder.decode(TokenResponse.self, from: data)
    tokenStorage.save(tokenResponse.accessToken)
}
```

**Updated `perform()` method** (two additions):
```swift
private func perform<T: Decodable>(_ request: URLRequest, decoding type: T.Type) async throws -> T {
    try await ensureAuthenticated()           // ← 1. register if no token yet

    var authedRequest = request
    if let token = tokenStorage.load() {
        authedRequest.setValue("Bearer \(token)", forHTTPHeaderField: "Authorization")  // ← 2. attach token
    }

    let (data, response) = try await session.data(for: authedRequest)
    guard let httpResponse = response as? HTTPURLResponse else {
        throw URLError(.badServerResponse)
    }
    if httpResponse.statusCode == 401 {       // ← 3. token rejected
        tokenStorage.delete()
        throw APIError.unauthorized
    }
    guard (200..<300).contains(httpResponse.statusCode) else {
        let detail = (try? JSONDecoder().decode(ServerErrorDetail.self, from: data))?.detail
            ?? HTTPURLResponse.localizedString(forStatusCode: httpResponse.statusCode)
        throw APIError.httpError(statusCode: httpResponse.statusCode, detail: detail)
    }
    return try decoder.decode(T.self, from: data)
}
```

**New `APIError` cases**:
```swift
case unauthorized
case missingRegistrationToken
```

**Why we need it**
The iOS client is the only entity that talks to the server. Without these changes every request would be rejected by the now-protected endpoints. The lazy `ensureAuthenticated()` approach means no explicit "login" step is needed in the UI — it happens automatically before the first real API call.

### Step 12 — Add Server section to `SettingsView.swift`

**File:** `iOS/ModelTraining/ModelTraining/ModelTraining/Views/SettingsView.swift`

**Short description**
Add a "Server" section to the Settings screen where the user configures the server URL, registration token, and can force a fresh registration.

**Simple explanation**
The user needs a way to type in the server address and the registration secret. This adds those two fields to the existing Settings screen, plus a button to clear the stored token and re-register (useful if the server's JWT secret is rotated).

**Detailed technical explanation**
1. Add `@EnvironmentObject var apiClient: GestureModelAPIClient` to the struct.

2. Add a new section at the top of the `Form`, before the existing "Camera" section:
```swift
private var serverSection: some View {
    Section("Server") {
        // Server URL
        HStack {
            Image(systemName: "network").foregroundColor(.blue).frame(width: 24)
            TextField("Server URL", text: Binding(
                get: { apiClient.baseURL.absoluteString },
                set: { if let url = URL(string: $0) { apiClient.baseURL = url } }
            ))
            .keyboardType(.URL)
            .autocapitalization(.none)
        }

        // Registration Token
        HStack {
            Image(systemName: "key.fill").foregroundColor(.orange).frame(width: 24)
            SecureField("Registration Token", text: $apiClient.registrationToken)
                .autocapitalization(.none)
        }

        // Re-register button
        Button("Re-register Device") {
            apiClient.clearToken()
        }
        .foregroundColor(.red)
    }
}
```

3. Add a `clearToken()` method to `GestureModelAPIClient` that calls `tokenStorage.delete()` so it can be called from the View without exposing `TokenStorage` directly.

4. Update `SettingsView_Previews` to add `.environmentObject(GestureModelAPIClient())`.

**Why we need it**
The registration token cannot be hardcoded — different users will deploy the server with different secrets. The Settings screen is the established place in the app to configure server-related parameters. Without this UI there is no way for the user to enter the registration token or recover from a token being invalidated.

## Verification Checklist

| Test | Expected result |
|------|----------------|
| `GET /examples/stats` with no token | `401 Unauthorized` |
| `POST /auth/register` with wrong `registration_token` | `403 Forbidden` |
| `POST /auth/register` with correct `registration_token` | `200` + JWT in body |
| `GET /examples/stats` with valid JWT | `200 OK` |
| `GET /health` with no token | `200 OK` |
| iOS: first upload after setting server URL + token | auto-registers, upload succeeds |
| iOS: tap "Re-register Device", then upload | re-registers, upload succeeds |
| Manually forge a JWT with wrong secret | `401 Unauthorized` |
